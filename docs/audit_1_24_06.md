## Binary-node code audit #1 (2026-06-24)

Adversarial review of the code written for the binary-node bootstrap, across four
components, each reviewed independently against the live source.

**Bottom line:** No critical bugs. **Everything that already ran on mainnet is trustworthy
as-run** — the snapshot-resorter's output, the migrate, the `preimages.rlp`, and the seed
were each independently confirmed correct (entry-count reconciliation, empirical
state/code matches, clean boot). The genuine risks are concentrated in the two things not
yet fully exercised: the catch-up driver's crash-safety, and Xatu data-completeness.

### What to fix before the next phase

1. **[HIGH] Catch-up resume corruption** — must fix before the 41k-block run on the new
   box (a multi-hour run *will* get interrupted). Resume from
   `max(latest, checkpoint_block())+1` and advance `forkchoice_update` incrementally
   alongside the trie's mid-loop flushes. Add RPC retry/backoff at the same time (a single
   transient error currently aborts the run straight into the corruption path).
2. **[HIGH] Verify Xatu completeness** — the downloader silently treats truncated/partial
   parquet as complete and conflates 404 with real failures, and the validator never checks
   that all ranges are present. Part of the ~1.78% coverage gap could be silently-missing
   populated ranges, not only the non-diff-source theory. Run a completeness pass (all
   25,341 files/table present **and** each actually parses) before trusting the gap's
   explanation.
3. **[MEDIUM] Quick hardening** — fix the stale `seed_head` doc comment (it still describes
   the old binary-root behavior — the exact bug already hit and fixed); add length guards in
   preimage-builder and a reconciliation guard in the resorter so an unhandled entry-type
   fails loudly instead of vanishing.

None of these affect the artifacts already produced; they harden the tools for reuse and
gate the not-yet-run catch-up.

### Catch-up driver — highest risk, never completed a block

Would produce a correct binary-trie chain on a clean, uninterrupted run on capable
hardware. The "needs per-block canonical head" concern proved **not** a bug — confirmed safe
(`find_parent_header` looks up by hash, `vm_db` reads the live in-place trie, and
`validate_state_root` is dead code for the binary trie). The break is purely the resume path.

- **[HIGH] Silent state corruption on resume** — `migrate.rs:362` (loop start), `:408`
  (deferred forkchoice). The loop resumes at `latest_block_number+1`, but `latest` is only
  advanced by the single `forkchoice_update` at the end — while the binary trie auto-flushes
  its checkpoint mid-loop (`flush_binary_trie_if_needed` every 128 blocks → persists
  `META_BLOCK_KEY = block_number`). After executing to block M and crashing before the final
  forkchoice: checkpoint = ~M (persisted), but `latest` = original head. On restart,
  `regenerate_head_state` sees `checkpoint(M) ≥ head` → "up to date" (no repair); the loop
  restarts at `latest+1` and re-applies blocks on top of M's already-advanced state → corrupt
  roots, no error raised. **Fix:** start from `max(latest_block_number, checkpoint_block())+1`,
  and/or call `forkchoice_update` incrementally so `latest`, canonical, and the checkpoint
  stay in lockstep.
- **[MEDIUM] No RPC resilience** — `migrate.rs:386-389`. Every `get_block_by_number` is a
  bare `?`; one transient error over ~41k sequential requests aborts the run — landing in the
  HIGH corruption path. Needs retry/backoff + per-block timeout.
- **[LOW] Finishes behind the moving tip; `finalized=target`; reload-barrier durability** —
  `migrate.rs:363-370, :409, :414-417`. Tip is sampled once; the chain advances ~7,200
  blocks/day, so a single run ends behind (needs an outer convergence loop or hand-off to the
  feeder). Marking the just-reached tip `finalized` is semantically wrong. Final durability
  hinges on `reload_binary_trie()` actually awaiting the async flush — worth a direct test.

### snapshot-resorter — ran on the live 150 GB snapshot

**Trustworthy as-run.** RLP read/write is value-preserving across every length class; the
partition+sort is correct; order-invariance holds. Entry counts reconcile exactly: 1
delete-marker + 395,142,821 accounts + 1,592,302,169 storage = the original total —
empirically proving zero silent drops on the actual input.

- **[MEDIUM] Unclassified add-op entries silently dropped; no reconciliation guard** —
  `main.rs:201-211`. An add-op entry that is neither 33-byte `'a'` nor 65-byte `'o'` is read
  but never counted or emitted. The only check (`emitted != n_stor`) is storage-only and
  wouldn't catch it. Safe on mainnet only because totals were manually verified. **Fix:**
  count every entry type and `bail!` on any unclassified add-op; assert
  `n_acct + n_stor + n_deletes == total_read`.
- **[LOW] Five defensive gaps** — `main.rs:140, 231/239, 182, 224-225, 134-143`. `cmd_verify`
  op-filter diverges from `cmd_resort`; phase-2 indexes a bucket without bounds checks
  (panics on truncation); output is non-atomic and not resumable; 256 `File::create().unwrap()`
  panic on FD/disk errors; per-bucket full load assumes hash-uniform distribution;
  `cmd_verify` loads the whole file into RAM.

### migrate: seed-head / seed-code / skipped-slots — all ran successfully

**Correct; no high/critical.** `seed_head` establishes a valid canonical head (checkpoint
META format verified against `BinaryTrieState::flush`; `open_store` bypass confirmed safe).
`seed_code` is correct and idempotent (writes both `ACCOUNT_CODES` + metadata). The
skipped-slots split correctly excludes zero-value slots (`!has_addr || !has_slot` only fires
on genuine gaps).

- **[MEDIUM] Stale / contradictory `seed_head` doc comment** — `migrate.rs:211-213`. The doc
  still says it "swaps in the binary-trie state root and stores it as the canonical head" —
  the old behavior. The code now correctly keeps the real MPT header. A future reader trusting
  the doc could re-introduce exactly the wrong-hash bug already hit and fixed. **Fix:** rewrite
  the doc to state it keeps the real header and that `--state-root` is informational-only.
- **[LOW] Caveats** — `migrate.rs:247-251, 285, 294, 592-600`. `seed_head` stores an empty
  `BlockBody` (checkpoint block reports 0 txs / no tx-by-hash; harmless). `seed_code` holds
  the whole ~14 GB code dump in RAM before writing (fine on 256 GB) and trusts the dump's code
  hash without re-keccak (empirically matched mainnet). `skipped_accounts` also counts
  RLP-decode failures but the report attributes all to "missing preimages" (inherited label).

### preimage-builder + Xatu scripts

**The `preimages.rlp` migrate consumed is trustworthy for the data it was fed** — keccak256,
20/32-byte length-tagging, hash-partition global sort, and gethdbdump framing are all correct
(confirmed: migrate resolved 388M accounts + 1.56B slots). The real exposure is upstream data
completeness, not the builder.

- **[HIGH] Downloader silently treats truncated/missing data as complete** —
  `download_xatu.py:29, 39-41`; `validate_parquet.py`. Skip logic is size-only
  (`exists and size>0 → skip`), and a failed curl removes the partial only if `size==0` — so a
  partial download with `size>0` is left in place and skipped forever (the earlier 6-corrupt
  mechanism, unfixed). A 404 (empty range) and a real failure both become `"miss"` with no HTTP
  status — so a populated range that fails to download is silently absent. The validator only
  checks files that exist and only probes trailing `PAR1`. **Fix:** verify the parquet footer
  before skipping; delete any non-zero-returncode output regardless of size; capture
  `%{http_code}` and treat only 404 as an acceptable miss; add a completeness pass asserting
  all 25,341 files/table present and parse. Run this before concluding the ~1.78% gap is purely
  non-diff sources.
- **[MEDIUM] preimage-builder: out-of-spec value length → silent truncation / OOB panic** —
  `preimage-builder/src/main.rs:101, 127-129`. Phase 1 writes `v.len() as u8` — a value >255
  bytes silently truncates the length byte. Phase 2 copies `vlen` bytes into a `[u8;32]` — any
  `vlen>32` panics. Didn't fire (clean 20/32-byte input) but unguarded. `write_rlp` correctness
  also rests on a `debug_assert!` compiled out in release. **Fix:** validate `vlen ∈ {20,32}`
  in phase 1; replace the debug_assert with a real length check.
- **[LOW] Encoding-assumption fragility & minor gaps** — the
  `unhex(substr(CAST(blob AS VARCHAR),3,N))` extraction is sound today (fixed-width, preserves
  leading zeros) but only because Xatu stores hex-ASCII in those columns; if it ever switches
  to raw-byte blobs this would silently corrupt every key (worth an output-row-count assert).
  Malformed hex → `unhex` NULL is swallowed by `DISTINCT` with no warning. `validate_parquet.py`
  writes `bad_parquet.txt` but the downloader won't re-fetch those unless manually deleted.
  `build_preimages.py` is a dead, would-OOM Python prototype — mark or delete.

### Cross-cutting themes

| Theme | Where | Note |
|---|---|---|
| Silent-failure over loud-failure | resorter (unclassified drop), downloader (truncation/miss), builder (length) | One-shot tools that assume clean input and don't reconcile/guard. Safe this run; add asserts before reuse. |
| Not crash-safe / not resumable | catch-up (corrupts), resorter (non-atomic), builder (partial bucket) | Only catch-up matters operationally (long multi-hour run); the others are short enough to re-run. |
| Trusted-input boundaries | seed-code (dump hash), xatu (blob encoding) | Acceptable given trusted sources; verified empirically. |
